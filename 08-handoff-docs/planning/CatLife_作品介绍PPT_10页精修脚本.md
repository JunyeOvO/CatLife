# CatLife 作品介绍 PPT 10 页精修脚本

日期：2026-06-29
目标：把初赛方案表达升级成复赛作品介绍 PPT，按官方要求结合实际作品效果截图，而不是只放设计概念。

## 1. 总体原则

- 页数目标：10 页左右，宁可少而硬，不堆长段文字。
- 每页必须有一个评审价值：创新性、应用价值、完成度或大模型应用能力。
- 至少 3 页使用真实工程截图或录屏截帧：主场景、小镇/猫、状态链。
- 大模型页必须讲清楚“为什么用、怎么用、隐私怎么保护、无网怎么降级”。
- 不能把 CatLife 讲成泛心理助手、泛宠物 App 或普通番茄钟。

## 3. 每页文案草稿

### 第 1 页：CatLife

主标题：CatLife
副标题：让专注自然发生，而不是被迫开始
底部：大学城今天吃什么 · AIGC 创新赛应用赛道复赛

讲述要点：

- 这不是一个更严厉的锁机工具。
- 它用一只会响应状态的猫，帮助用户从分心自然过渡到专注。

### 第 2 页：问题不是不会计时

页面三句话：

1. 很多学生不是不知道要学习，而是很难从碎片化手机操作中停下来。
2. 最困难的不是计时开始后，而是从分心进入专注的前 30 秒到 3 分钟。
3. CatLife 把这个阶段定义为“专注过渡阶段”。

建议图：

- 左：高频刷动/切换；
- 右：学习任务；
- 中间：断裂的过渡桥。

### 第 3 页：现有工具的缺口

| 工具 | 解决方式 | 问题 |
|---|---|---|
| 番茄钟 | 用户主动开始计时 | 假设用户已经准备好专注 |
| 强锁机 | 限制退出 | 容易削弱自主性，引发抵触 |
| 宠物 App | 情感陪伴 | 容易变成新的互动娱乐 |
| CatLife | 识别过渡状态，用猫降低刺激 | 关注进入专注前的关键阶段 |

### 第 4 页：我们的解法

主句：

```text
识别行为节奏变化，让猫咪承接注意力，再用轻锁定降低干扰。
```

流程：

```text
本地交互特征 -> 专注分/活跃分/分心分 -> 四状态猫咪 -> UI 降干扰 -> 奖励复盘
```

### 第 5 页：四状态体验

| 状态 | 用户行为 | 猫咪反馈 | UI |
|---|---|---|---|
| 普通 | 高频点击/浏览 | 活跃、看向用户 | 完整互动 |
| 过渡 | 操作减少/停顿 | 靠近、嗅闻、倾听 | 弱提示 |
| 专注 | 低频稳定 | 安静趴下/呼吸 | 轻锁定 |
| 奖励 | 完成会话 | 开心尾巴/庆祝 | 成长反馈 |

必须换成真实录屏帧，不要只放表格。

### 第 6 页：3D 作品完成度

可写：

- 已制作 10 个猫咪动作，覆盖 idle、观察、嗅闻、倾听、摇尾、伸懒腰等。
- 已整理猫咪小镇当前 no-merge 场景，避免破坏材质和 UV。
- 已形成 Unity MVP 交接资产，下一步是正式工程导入和 Android 真机验证。

注意：不能写“小镇已进入正式 Unity 主场景”，除非后续实际验证完成。

### 第 7 页：技术架构

页面左侧架构：

```text
应用内事件 -> 去标识化聚合 -> 本地规则评分 -> 猫咪状态机
                                  |
                                  v
                         可选 LLM 生成陪伴文案
```

页面右侧说明：

- 不读取输入原文；
- 不录屏；
- 不跨 App 监控；
- API 只接收聚合摘要；
- 无网络时用本地模板。

### 第 8 页：隐私与可运行性

这里要服务评审的“可行性”和“完成度”：

- APK 可运行证据：填入真机截图/录屏二维码/日志摘要；
- 隐私边界：只采集本 App 内交互节奏；
- 质量门禁：安装、启动、状态链、无崩溃。

### 第 9 页：应用价值与验证

建议放 5 份访谈摘要：

| 用户 | 场景 | 主要痛点 | 对 CatLife 的反馈 |
|---|---|---|---|
| U1 | 图书馆复习 | 开始前刷手机停不下来 | 希望有温和提醒 |
| U2 | 宿舍学习 | 强锁机会焦虑 | 轻退出更可接受 |
| U3 | 考前背书 | 专注中被通知打断 | 希望界面更安静 |
| U4 | 自习室 | 番茄钟开始难 | 猫咪过渡有吸引力 |
| U5 | 写作业 | 需要完成后反馈 | 奖励记录有动力 |

这些是模板，必须替换为真实访谈或问卷结论后再进终版 PPT。
