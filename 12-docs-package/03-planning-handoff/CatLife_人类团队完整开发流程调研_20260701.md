# CatLife 人类团队完整开发流程调研

日期：2026-07-01

目标：把 CatLife 从当前本地 3D 资产、UI 装配包和规划文档，推进到可运行 APK、演示视频、PPT、海报和代码包的完整比赛提交。本文按“真实人类团队会怎么做”拆解，不假设一步到位，也不把 AI/脚本输出当作免验收结果。

## 0. 调研依据

### 官方与行业依据

| 来源 | 对 CatLife 的影响 |
|---|---|
| Unity Android Build Process：Unity 可直接构建 Android，也可导出 Android/Gradle 工程；Android 不可用时需先安装 Android Build Support。https://docs.unity3d.com/Manual/android-BuildProcess.html | 确定 APK 路线、Unity 构建链和导出链 |
| Unity Android Build Settings：Android 构建配置包含纹理压缩、平台设置等。https://docs.unity3d.com/6000.1/Documentation/Manual/android-build-settings.html | 确定 ASTC/ETC2、ARM64、包名、版本号等 |
| Unity Model Import Settings：FBX/模型导入时要定义 Rig、Animation Clips；同骨架动画可复用。https://docs.unity3d.com/6000.4/Documentation/Manual/class-FBXImporter.html | 确定猫咪动画导入和 Animator 验收方式 |
| Unity Generic Animations：非人形角色应按 Generic 动画体系导入。https://docs.unity3d.com/6000.4/Documentation/Manual/GenericAnimations.html | 猫是四足非人形角色，应优先 Generic Rig |
| Unity Profiler：性能应连接到目标平台应用采集，而不是只看 Editor。https://docs.unity3d.com/Manual/profiler-profiling-applications.html | 性能验收必须包含真机/目标平台证据 |
| Unity URP Performance：移动端要减少阴影距离、额外灯光阴影、相机数量、MSAA 等成本。https://docs.unity3d.com/6000.2/Documentation/Manual/urp/configure-for-better-performance.html | 确定小镇渲染和后处理约束 |
| Unity Mobile Optimization：移动端应减少材质数量、使用图集、控制实时像素光等。https://docs.unity3d.com/2019.4/Documentation/Manual/MobileOptimisation.html | 确定小镇材质、贴图、批处理优化方向 |
| Unity Skybox：天空盒应通过 Lighting/Environment 或相机 Skybox 控制，并可匹配 Directional Light。https://docs.unity3d.com/6000.1/Documentation/Manual/skyboxes-using.html | 最终天空建议在 Unity 做，不在 Blender 烘死成几何体 |
| Blender glTF 2.0：glTF 适合传输/加载 3D 模型，动画导出受 Actions/NLA 等设置影响。https://docs.blender.org/manual/en/latest/addons/import_export/scene_gltf2.html | 小镇/动画导出可选 glTF，但动画要明确导出策略 |
| Blender FBX：FBX 广泛用于 3D 数据交换，尤其动画角色和复杂场景。https://docs.blender.org/manual/en/latest/addons/import_export/scene_fbx.html | 猫动画/Unity 导入优先考虑 FBX |
| Blender Color Management：颜色管理是场景级设置，影响渲染和多设备显示效果。https://docs.blender.org/manual/en/latest/render/color_management/index.html | Blender 预览图不能直接等同 Unity/手机最终色彩 |
| Android Studio Profiler：性能问题需要分析 CPU、内存、图形和电量使用。https://developer.android.com/studio/profile | 真机 QA 要记录 CPU/内存/图形/电量风险 |
| Android vitals：关注稳定性、性能、电量、权限等质量指标。https://developer.android.com/topic/performance/vitals | APK 可提交前要排查 crash、ANR、慢帧、电量等 |
| Android startup：Android vitals 将冷启动 5 秒及以上视为过慢。https://developer.android.com/topic/performance/vitals/launch-time | CatLife 冷启动目标应低于 5 秒 |
| ADB screenrecord：可录制设备屏幕 MP4，默认最长 3 分钟且不录音。https://developer.android.com/tools/adb#screenrecord | 演示视频真机素材可用 ADB 录屏，但旁白需后期配 |

## 1. 人类团队会先定的原则

| 原则 | CatLife 具体执行 |
|---|---|
| 可运行优先 | APK 能安装启动、看到猫和小镇、完成一次专注闭环，是 P0 |
| 本地最新资产为准 | Blender 当前可用小镇和猫动画是源头，旧 Unity 场景只做参考 |
| 每步留证 | 每个阶段必须有截图、录屏、日志、manifest 或性能表 |
| 不合并破坏资产 | 小镇不要再全场景合并网格；保留材质、UV、硬边和对象层级 |
| UI 叠加于场景 | 无 UI 时画面只有猫 + 小镇；UI 是 Canvas Overlay |
| 隐私优先 | 行为识别只用聚合特征，不录屏、不读文本、不跨 App 监控 |
| 真机结论高于 Editor | Editor 通过不等于 APK 可提交，最终看 Android 真机 |

## 5. Phase 2：Blender 美术收口

### 小镇处理顺序

1. 打开当前小镇源文件。
2. 确认底座、主建筑、入口、奖励树、猫窝、鱼店、围栏都在正确集合。
3. 清理明显失效对象：隐藏测试相机、废弃天空几何、未使用生成物。
4. 调整材质：草地黄绿、侧壁暖棕、建筑低反光、石板暖米色。
5. 不做全场景合并；只按模块导出或保持对象层级。
6. 输出预览图：全景、正面猫视角、入口、奖励树。
7. 导出 Unity runtime 格式。

### 天空盒处理结论

| 场景 | 推荐做法 |
|---|---|
| Blender 出预览图 | 可以临时加半球/天空板/云片，但必须可删除 |
| Unity 正式 App | 用 Unity Skybox/Lighting Environment 或相机 Skybox，不建议把天空盒几何烘死进小镇模型 |
| 当前状态 | 已按要求另存 no-skybox 小镇版，后续天空在 Unity 做更稳 |

Unity 官方建议通过 Lighting/Environment 设置 Skybox Material，并让 Directional Light 与天空光源方向一致；这比在 Blender 里把碗状半球作为模型导入更符合运行时工作流。

### 导出检查

| 检查项 | 标准 |
|---|---|
| 坐标 | Blender 保持 `+Z` up；Unity 导入器处理坐标转换 |
| 比例 | 以猫与门、长椅、路面关系肉眼合理为准 |
| 材质 | 不透明材质使用 Opaque；透明材质单独检查黑边 |
| 法线 | 低模硬边保留，不批量 Smooth |
| 毛刺 | 禁止用“全场景合并”解决；先查重叠面、透明、z-fighting |
| 文件 | 导出后记录路径和哈希 |

## 8. Phase 5：猫模型和动画导入

### 导入原则

猫是四足角色，不是 Humanoid。按 Unity Generic 动画流程导入更合理。

| 项 | 操作 |
|---|---|
| Rig | Animation Type = Generic |
| Scale | 与小镇门、路面、镜头对齐 |
| Rotation | 头朝主相机可控方向，不再改骨骼源 |
| Clips | 拆出 Idle、Walk、Sniff、HeadTilt、Focus、Reward 等 |
| Animator | 用状态参数驱动，不把 UI 逻辑写进 Animator |

### 最小 Animator

```text
Normal: IdleBreath
Transition: CuriousSniff / HeadTiltListen
Focus: IdleBreath / LowMotionFocus
Reward: TailWagHappy
```

### 通过门槛

- 猫脚贴地，不飞、不陷。
- 头尾方向正确。
- 4 个状态能通过按钮或调试面板切换。
- 用短录屏证明动画不是静态模型。

## 9. Phase 6：摄像机、灯光和天空

### 摄像机

| 控制 | 规则 |
|---|---|
| 初始视角 | 猫在前景，小镇背景占主要画面 |
| 旋转 | 水平 360 度原地旋转，高度不变 |
| 移动 | 前后水平移动，高度不变 |
| 禁止 | 顶视角切换、自由俯仰、roll、上下飞行 |

### 天空和灯光

| 内容 | 建议 |
|---|---|
| Skybox | Unity 里做 Procedural/Skybox Material，不把 Blender 半球作为正式导入资产 |
| Sun Source | Directional Light 与天空明亮方向一致 |
| Fill | 移动端用环境光/烘焙/少量 fill，不堆实时灯 |
| Shadows | 控制阴影距离和级联数量，小物体可关闭投影 |
| Post-processing | Bloom/Color Grading 可轻量使用，避免重景深 |

### 通过门槛

- 首页截图接近参考图：亮天、暖色、低多边形、猫和小镇清楚。
- 手机上不因天空/后处理明显掉帧。

## 11. Phase 8：状态机和行为识别

### 状态链

```text
Normal -> Transition -> Focus -> Reward -> Normal
                 └-> DistractionNudge -> Focus
```

### 模块划分

| 模块 | 责任 |
|---|---|
| `FocusStateMachine` | 状态、事件、时长 |
| `BehaviorSignalProvider` | 聚合行为信号，MVP 可模拟 |
| `CatAnimationDriver` | 状态到 Animator |
| `CatChatBubbleController` | 状态到短文案 |
| `FocusSessionController` | 开始、退出、完成、奖励 |
| `LocalRecordStore` | 本地会话记录 |

### 行为信号边界

| 可以用 | 不应该用 |
|---|---|
| 本 App 内点击频率、停留时长、专注时长、中断次数 | 录屏、读取输入内容、跨 App 监控、通讯录、短信 |

### 通过门槛

- 手动/模拟信号能跑完整链。
- 状态切换有日志。
- 记录页能看到至少一条会话。

## 13. Phase 10：移动端优化

### 先测再改

人类团队不会凭肉眼说“应该能跑”。流程是：

```text
Unity Stats -> Unity Profiler on device -> Android Studio Profiler -> logcat/crash/ANR
```

### 优化顺序

| 优先级 | 检查 |
|---|---|
| P0 | 关掉不必要实时阴影、减少相机、减少实时灯 |
| P0 | 贴图压缩，Android 优先 ASTC，不兼容再 ETC2 |
| P0 | 合并材质思路用图集和 sharedMaterial，不全场景合并网格 |

## 18. 你可以自行开发的关键步骤

如果你要自己动手做关键步骤，优先顺序如下：

1. Unity 新建干净工程并导入小镇底座 + 主建筑。
2. 建立 `CatLifeTownRoot` Prefab 和主相机。
3. 导入猫 FBX，设置 Generic Rig 和 Animator。
4. 先做 4 个状态按钮切动画，不急着接真实行为识别。
5. 接 UI 装配包，主场景只保留必要按钮。
6. 接猫咪气泡，验证状态到文案。
7. 做 Android Development Build，先确认真机能跑。
8. 再做 LLM 真实调用和隐私说明。
9. 最后做性能优化、视频、PPT、海报。

## 19. 失败时的排查顺序

| 问题 | 排查顺序 |
|---|---|
| Unity 场景黑屏 | Camera -> Scene in Build -> Root active -> Console error |
| 小镇不见 | 模型导入路径 -> scale -> clipping plane -> material shader |
| 猫飞起 | Rig/root transform -> Animator root motion -> prefab y 坐标 |
| 动画不切 | Animator state name -> parameter -> controller reference |
| UI 遮挡 | Canvas scale -> safe area -> anchor -> sorting |
| 草/石头漂浮 | Blender 采样面 -> 顶面高度 -> pivot -> Unity scale |
| 黑线毛刺 | 透明材质 -> z-fighting -> backface -> shadows -> mipmap |
| APK 装不上 | ABI -> minSdk -> signing -> package conflict |
| 真机卡顿 | 关阴影 -> 降贴图 -> 减材质 -> 查 Profiler |
| LLM 失败 | Mock fallback -> timeout -> key/env -> 网络权限 |

## 20. 最小可提交定义

只有满足以下条件，才算 CatLife 可以进入比赛提交：

- APK 可在至少一台 Android 真机安装启动。
- 主界面第一眼看到猫和猫咪小镇。
- 用户能点击开始专注。
- 猫至少有 Normal、Focus、Reward 三类可见状态。
- 至少一次完整录屏展示核心闭环。
- 有 PPT、视频、海报、APK、代码包五项提交物。
- 代码包明确标注大模型调用或端侧 SDK 接入点。
- 有 logcat、构建日志、截图/录屏、SHA256 证据。

## 21. 当前下一步

按人类团队流程，当前最合理的下一步不是继续扩展文档，而是：

1. 在 Unity 中新建或确认干净工程。
2. 导入当前 no-skybox 小镇版本或从 Blender 导出 runtime FBX/GLB。
3. 在 Unity 中做 Skybox/Directional Light，而不是在 Blender 中烘死天空盒。
4. 放入猫模型，先做摄像机 + 猫 + 小镇的一屏真机目标画面。
5. 再接 UI、状态机和 Android 构建。
