## 一、核心原则

### 1.1 质量压倒一切
- 复赛评分中"作品完成度"占20%，是初赛不存在的全新维度
- 5项材料缺任何一项直接淘汰
- **Demo必须完备**：评委点名批评"工程化不足"

### 1.2 差异化是第一竞争力
- 创新性占40%权重
- 必须清晰阐述：为什么CatLife不是另一个番茄钟/锁机App/宠物App
- 核心差异：AI行为识别→猫咪状态反馈→轻锁定引导，形成闭环

### 1.3 大模型必须真调用
- "通过API调用实现产品功能"是硬性要求
- 最低标准：至少一个API调用场景可演示
- 代码包必须标注大模型调用部分

```
2. 痛点 — 4个核心痛点(开始难/持续难/回归难/工具弱)

6. 产品架构 — 四状态闭环图 + 交互流程
```

## 五、大模型集成规范

### 5.1 双模型混合架构（推荐策略）

> 基于复赛交流会信息：vivo提供蓝心3B端侧模型 + 不限制使用任何第三方API

| 层级 | 模型 | 场景 | 优势 |
|------|------|------|------|
| **端侧** | 蓝心3B (vivo提供) | 猫咪实时对话、敏感内容过滤 | 低延迟、隐私安全、离线可用 |
| **云端** | DeepSeek-V3 (或豆包) | 专注状态分析、专注总结 | 高精度推理、复杂语义理解 |

**关键理由**（可用于PPT/答辩）：
1. 隐私保护：用户行为数据和猫咪对话不上传云端 → 端侧处理
2. 实时体验：猫咪反馈需要<500ms延迟 → 端侧推理
3. 复杂分析：专注模式判定需要深度语义理解 → 云端大模型
4. 成本优化：高频对话用端侧（零成本），低频分析用云端（Token可控）

### 5.2 端侧集成方案（蓝心3B）

```java
// 蓝心3B SDK调用示例（极简三步）
// 1. 初始化
LLMEngine engine = new LLMEngine(context);
engine.init();

// 2. 喂Prompt → 流式返回
engine.generate("用一只猫的口吻鼓励用户继续专注，回复<30字",
    new LLMCallback() {
        @Override
        public void onToken(String token) { /* 逐字显示 */ }
        @Override
        public void onComplete(String full) { /* 完成 */ }
    });

// 3. 释放
engine.release();
```

**注意**：
- 蓝心3B是纯文本模型，不支持多模态
- 已内置在云真机系统（X300 Pro, Android 16）
- 调通后在真机上也直接可用
- 接入方式是Java/C++ SDK，非HTTP API
- 官方提供Demo APK源码，可直接改造

### 5.3 可落地场景（按P0-P2排）

| 优先级 | 场景 | 输入→输出 | 调频 |
|--------|------|-----------|------|
| **P0** | 智能专注判定 | 行为特征→state+score | 每30s |
| P1 | 猫咪对话 | 用户文本→简短回复(<50字) | 按需 |
| P2 | 专注总结 | 行为数据→人性化小结 | 专注结束时 |

### 5.2 API选型推荐

| 模型 | API | 成本 | 推荐 |
|------|-----|------|------|
| DeepSeek-V3 | 开放API | 极低 | ★★★（首选） |
| 豆包lite | 火山引擎 | 学生免费 | ★★★（备选） |
| Qwen2.5 | 阿里/本地 | 免费 | ★★ |

### 5.3 调用规范

```csharp
// Unity C# → HTTP POST → API
// LLMClient.cs 统一封装
// Prompt模板：system+user → JSON response
// 错误处理：超时3s → 降级到规则引擎
```
